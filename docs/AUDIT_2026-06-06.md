# Report Audit — nc-calendar (2026-06-06)

> Audit multi-agente (33 agenti) con verifica adversariale + grounding su Sentry.
> 50 finding grezzi → 22 confermati (3 falsi positivi scartati).
> Conteggi: 🔴 0 · 🟠 3 · 🟡 6 · 🟢 ~16 · ℹ️ 7.

## Riepilogo esecutivo

App **solida e ben difesa in profondità**: auth verifica il JWT, service_role non esposta, segreti oscurati, edge function validano ruolo+proprietà, consumo crediti atomico/race-safe, anti-wipe calendario funzionante. Sentry: **nessun errore reale** in produzione (4 eventi = test). Difetti concreti: 1 sicurezza alto (coach può manomettere sessioni di altri coach), 2 bug alti (rimborso fantasma su import Google, modifiche coach non sincronizzate su Google), + medi/bassi di coerenza e pulizia.

## 🟠 Alti

- **A1 — IDOR cross-coach** (`supabase/migrations/20260606123152_*.sql:16-28,102-113`): `reconcile_gcal_cancel`/`reconcile_gcal_move` controllano solo "è un coach", non la proprietà → un coach può annullare/spostare/rimborsare sessioni di altri coach. *Fix:* aggiungere check `coach_id = caller` (come fanno le funzioni gemelle). **[migration]**
- **A2 — Rimborso fantasma su import Google** (`src/lib/gcal.functions.ts:817-878` + `cancel_booking`/`mark_booking_special`/`reconcile_gcal_cancel`): le sessioni importate non consumano crediti, ma cancellandole il rimborso scala comunque un credito da un'altra sessione → saldo corrotto. *Fix:* non rimborsare se `google_event_id` valorizzato, o colonna `credit_consumed`. **[migration]**
- **A3 — Modifiche coach dal profilo cliente NON sincronizzano Google** (`src/routes/trainer.clients.$id.tsx:585-659`): `saveBookingEdit` cambia DB ma non aggiorna/cancella l'evento Google → promemoria sbagliati / eventi fantasma per il cliente. *Fix:* dopo l'UPDATE chiamare `gcalUpdateEvent`/`gcalDeleteEvent`. **[codice]**

## 🟡 Medi

- **M1 — Cliente può estendere `end_at`** (`20260522100000_*.sql:86-106`): whitelist trigger non copre `end_at` → griefing slot. *Fix:* aggiungere `end_at` alla whitelist o forzare reschedule via RPC. **[migration]**
- **M2 — `reschedule_booking` doppio consumo in casi limite** (`20260606071612_*.sql:97-147,155-188`): incrementa nuova allocazione anche se il rilascio non trovato. *Fix:* fail-closed. **[migration]**
- **M3 — "Regole di prenotazione" salvate ma ignorate** (`trainer.availability.tsx:478`, `booking-slots.ts:145`, `client.book.tsx:214`): preavviso/orizzonte/buffer hardcoded. *Fix:* collegarle o disabilitare la card. **[codice]**
- **M4 — Due flussi reschedule cliente, notifica coach incoerente** (`reschedule-drawer.tsx:189`, `client-reschedule-sheet.tsx:204`): dalla dashboard il coach non viene avvisato. *Fix:* unificare + notifica nella mutation condivisa. **[codice]**
- **M5 — Dettaglio cliente: query ridondanti** (`trainer.clients.$id.tsx:179-334`): profiles letto 2×, bookings 2×. *Fix:* unire + parallelo. **[codice]**
- **M6 — 2 letture Google a ogni apertura calendario** (`trainer.calendar.tsx:86-118` + `calendar-gcal-review.tsx:51-60`): pannello revisione non throttlato. *Fix:* riusare dati / stesso throttle. **[codice]**
