## Strategia generale di attacco

Regole d'oro per evitare regressioni:

1. **Una PR per gruppo tematico**, non per singolo finding (riduce overhead push/merge per l'utente non-dev). Esempio: tutti i token color in 1 PR, tutti i bug `as any` in 1 PR.
2. **TSC verify prima di ogni commit** — nessuna eccezione. Solo i 2 errori Sentry pre-esistenti sono accettabili.
3. **CRIT → HIGH → MED → LOW**: priorità decrescente. CRIT sono sicurezza/data-loss e vanno fixati subito, MED/LOW si possono accumulare e droppare in batch.
4. **Mai allargare lo scope mid-PR** — se trovo un altro bug mentre fixo, lo aggiungo all'audit e lo affronto separatamente.
5. **Per fix UI rischiosi**: dopo Lovable update chiedere all'utente test visivo prima di passare al fix successivo.
6. **Mai rimuovere "dead code" senza grep cross-check** — un componente potrebbe essere referenziato da lazy import / dynamic route.

## CRIT — Sicurezza / Data integrity (2)

### CRIT-1 · sync-calendar: token leak in catch-all error
- **File**: `supabase/functions/sync-calendar/index.ts:1284`
- **Problema**: il catch-all `return { error: error.message }` può ritornare risposte Google OAuth contenenti `refresh_token` / `access_token`. La sanitizzazione applicata al refresh flow (riga 89-104) non copre questo path.
- **Strategia**: wrappare il `catch` finale con la stessa funzione di scrubbing già esistente (cercarla e riusarla) oppure introdurre un helper `sanitizeError(e)` condiviso. Mai più di una riga toccata in produzione.
- **Rischio**: basso — solo lato server, no UI impact.
- **Verifica**: tail dei Lovable logs dopo deploy + manually trigger un sync con un token expired per confermare che la response non contiene secret.
- **Status**: `[x]` — fixato in commit `3fd7312` (branch `fix/crit-edge-functions-security`, mergiato in main via PR #11). Helper `sanitizeUpstreamError(e)` aggiunto sotto `json()`.

### CRIT-2 · booking-notifications: WhatsApp phone senza validazione
- **File**: `supabase/functions/booking-notifications/index.ts:264`
- **Problema**: `client_phone` viene passato all'API WhatsApp senza length cap o regex E.164. Input malformato → potenziali errori 4xx ripetuti / abuse vector se l'attaccante può iniettare phone arbitrari.
- **Strategia**: aggiungere helper `isValidPhone(s)` (regex `^\+?[1-9]\d{7,14}$`) prima della chiamata WhatsApp. Se invalid → skip + log warning, non lanciare exception.
- **Rischio**: basso.
- **Verifica**: deploy + manually trigger una notification con phone vuoto/malformato → non deve crashare l'edge function.
- **Status**: `[x]` — fixato in commit `3fd7312` (stesso branch/PR di CRIT-1). Helper `normalizeAndValidatePhoneE164(raw)` + skip + log warning sul `bookingId` se phone invalid.

REVOKE da PUBLIC/anon/authenticated.

### HIGH-2 · Bookings query senza `.limit()`
- **File**: `src/lib/queries.ts:200-220`
- **Problema**: `selectBookingsByCoach` / `selectBookingsByClient` non hanno cap. Coach con 1000+ bookings storici fetcha tutto al mount → CLS + memoria.
- **Strategia**: aggiungere `.limit(500)` come safety net + `.order("scheduled_at", { ascending: false })` esplicito. Se servono record più vecchi → query separata "load more" lazy.
- **Rischio**: medio — potrebbe rompere lo storico Timeline se utente ha più di 500 bookings (improbabile a oggi).
- **Verifica**: testare la timeline cliente storica dopo deploy.
- **Status**: `[x]` — branch `fix/db-query-growth-high-1-2`. Costante `BOOKINGS_FETCH_LIMIT = 1000` + `.order("scheduled_at", { ascending: false })` + `.limit(...)` su `selectBookingsByCoach` e `selectBookingsByClient`. Limit a 1000 (più alto dei 500 dell'audit) perché un cliente con percorso pluri-annuale + booster può arrivare a ~672 booking teorici; 1000 garantisce margine. Cap è circuit-breaker, non paginazione UI (consumer fanno filtering downstream).
